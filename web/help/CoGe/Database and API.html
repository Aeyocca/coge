How is the CoGe genomes database organized?

We have made great efforts to keep the design of the database for storing and accessing genomic information simple while not compromising performance. The core part of the database (those tables that store genomic information specifically) consists of 13 tables. The central concept around which all other information orbits is genomic "features". A genomic feature merely refers to some genomic region to which additional information is attached. Take a "gene" as an example. Although the specific definition of a gene varies, abstractly it refers to a genomic region that contains biologically functional information. This information, when read through a process called transcription, gives rise to the creation of RNAs. These RNAs either carry out some cellular function (such as tRNAs and rRNAs) or encode information for use in creating proteins through the process of translation. Thus, in order to track a region of a genome that has associated additional information, all we need to know is its location. If you look at the schema for the CoGe genome database, you will see a relationship between the feature table and the location table. The feature table stores only three types of information, the internal database id of a feature (each feature gets a unique id), the feature_type_id which links a feature to the feature_type table (although you can probably guess the information stored in that table, we will go into more detail of the feature_type table further on in this discussion), and the data_information_id (which is the dataset from which the feature was obtained). The location table stores the information about the position of the feature: the chromosome from which it is derived, the start position (nucleotide coordinates), the end position, the strand (+/1 for top strand, -/-1 for bottom strand), and the feature database id (feature_id) to which the location belongs. The relationship between the feature and location tables is a "one to many" relationship where one feature can have more than one location associated to it. The reason for this will become clear in a moment.

Some genes contain information for transcribing more than one RNA where different RNAs use different genomic sequence contained within the gene. In order to track this kind of information (aka alternative splicing), we need to specify additional genomic features for each RNA. However, since not all of the DNA sequence of the gene is necessarily used for a given RNA, we may need to track different genomic locations for the gene and each RNA. To do this, we can specify a unique feature for the gene and each RNA, and specify the appropriate genomic location(s) used for each. By having a one to many relationship between the feature and location tables, we can accommodate all this information. For example:

GeneA: Chromosome 1, Strand 1, Position (1000-10000)
RNA_A1: Chr 1, Strand 1, Position (1100-3000,4000-5000,8000,-9500)
RNA_A2: Chr 1, Strand 1, Position (1100-3000,4000-6000,7000-7500,8000,-9500)

To extend this example further, assume that each RNA listed above encodes a protein. However, not all of the information that is transcribe into RNA is necessarily translated into protein. In this case, we can specify additional features for just the protein coding region using the same process that was done for the RNA coding regions. Likewise, the same can be done for cis-acting sequences that act to regulate the genes transcriptional activity.

One thing that you may have noticed from this example is that we specified several different kinds of genomic features: genes, RNAs and proteins coding regions (aka CDS). However, it is easy to find several more examples of different types of genomic features. For example, there are several different types of RNA genes besides those that code for proteins (aka mRNA): tRNA, rRNA, micro RNA, snRNA, snoRNA, etc. Additionally, there are several genomic features that don't code for RNA, but are useful to track because they have function. A TATA-box is such an example of a genomic feature. TATA-boxes are genomic regions that contain the information needed to position RNA polymerases (usually RNA pol II) correctly in order to transcribe a gene in eukaryotic transcription systems. Another example that is similar to TATA-boxes are conserved non-coding sequences (CNSs) which have been recently identified in Arabidopsis (Thomas et al., 2007).

In order to track this kind of information (different types of features), there is a table in the CoGe genomes database designed just for that purpose: the feature_type table. This table has three types of associated information: the internal database id of the feature type, the name of the feature type (e.g. mRNA), and the description of the feature type (usually not used, but good to have just in case.)

One problem with genomic features is the plethora of alternative names that may be used to refer to a single feature. To accommodate this, there is a table in the database called feature_name. Each entry in this table contains a name and description (optional) for a feature and is linked to the feature table with a one-to-many relationship (i.e. one feature entry can have multiple feature_name entries.)

At this point, we have specified the central concept of the CoGe genomes database: the genomic feature. A feature needs to be of a particular type (feature_type table), has genomic coordinates (location table), and one or more names (feature_name table). In addition to this information, there are additional kinds of information that can be associated with a feature. Generally, this additional information breaks into type types: specific sequences (e.g. translated protein sequence) and annotations. If a feature has a specific non-DNA sequence associated with it, such as a protein sequence, this information can be stored in the sequence table. The sequence table in turn is associated with the sequence_type table to store the name and description of the type/kind of sequence.

The term "annotation" is a broad category of associated information about a gene that is gathered through experimental and computational analysis of a genomic feature. Some types of annotations are functional information, database references, literature references, cellular locations, etc. In order to track this kind of information in a flexible yet organized manner, a three level annotation schema is employed. At the root is the annotation table which stores an annotation, the feature to which it belongs, and the type of annotation (stored in the annotation_type table) that it is. In order to organize types of annotations, there is an annotation_type table that has the name and description of the type of annotation. Additionally, annotation_types can be grouped into broader categories using the annotation_type_group table. This tree-tiered approach allows maximum flexibility for specifying annotations, types of annotations, and groups of types of annotations.

An example of this three-tiered annotation approach is well suited for Geneontology annotations. Let us say we have the following annotation for a gene:

go_function: nucleic acid binding [goid 0003676].

For this annotation, we can create an annotation_type_group with the name of "go_function", an annotation_type with the name of "0003676" and description "goid", and the annotation will be "nucleic acid binding". Using this type of a schema, it makes finding all annotation_types with an annotation_type_group of "go_function" trivial. Likewise, it because easy to find all features with an annotation_type of "0003676".

The next major section of the CoGe database is for storing information about a dataset, the organism to which it belongs, and the actual genomic sequence. The key table for this information is the data_information table. Although this table should have been called the dataset table, the primary information this stores is the name, description, and version of a given dataset. Though it the data_source, feature, organism, and genomic_sequence tables are linked. The data_source table stores the name, description and web-link of a source providing genomic data (e.g. genbank, JGI, TIGR, Ensembl). The organism table stores the name and description of an organism (e.g. Arabidopsis thaliana, Escherichia coli); the genomic_sequence table stores the genomic sequence. Since genomes can be large (the human genome is ~3,000,000,000 base pairs), genomic sequence is broken into chunks (usually 10,000 nucleotides).

Using this schema to store genomic data allows for storing genomes in any number of configurations. For example, a genome (and all chromosomes contained therein) can be provided as a single file or multiple files (e.g. one chromosome per file). Also, multiple versions of a genome can be stored by specifying the "version" column of the data_information table. And finally, additional feature/annotation files can be stored that are associated with a version of a genome, but do not actually contain genomic sequence (e.g. an annotation file of all protein domains in a genome).
